# Implementation Plan: Voice Clone Synthesis

**Branch**: `feature/time-machine-voice-clone-synthesis` | **Date**: 2026-07-15 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/005-voice-clone-synthesis/spec.md`

## Summary

Harden the existing local Qwen3-TTS/MLX voice-clone path around its already-working Base-model integration. Introduce typed codec-token progress at the model's documented 12.5 Hz rate, validate candidate audio before acceptance, snapshot request inputs, discard superseded asynchronous results, and commit WAV output through exclusive staging. Preserve both accepted synthesis and alteration if a same-input regeneration fails. Keep the expensive end-to-end smoke test opt-in and add fast pure contract coverage.

## Technical Context

**Language/Version**: Swift 6.2

**Primary Dependencies**: SwiftUI, Observation, AVFoundation, MLX, `swift-qwen3-tts`, Swift Testing

**Storage**: Application Support session WAV files and in-memory finite Float32 mono samples

**Testing**: Swift Testing with Xcode; gated Qwen3-TTS smoke test

**Target Platform**: macOS 26.0+, Apple Silicon

**Project Type**: Native macOS SwiftUI application generated by XcodeGen

**Performance Goals**: Dedicated serial generation executor; throttled UI progress; no MainActor blocking; throughput displayed after completion

**Constraints**: Local-only inference; Base model only; 24 kHz reference WAV of at least three seconds; backend generation is synchronous and not reliably cancellable

**Scale/Scope**: One active synthesis, one staged candidate WAV, one accepted synthesis, one accepted alteration

## Constitution Check

### Initial Gate

- **Local-only privacy**: Pass. No remote synthesis path is introduced.
- **Apple Silicon execution**: Pass. Existing MLX/Qwen3-TTS integration remains.
- **Explicit contracts**: Pass. Request, progress, result, validation, and errors stay typed.
- **Responsive UI**: Pass. Blocking model work remains on a dedicated serial executor.
- **Transactional media**: Pass. Candidate output is staged and atomically committed.
- **Testability**: Pass. Progress, statistics, output validation, and commit semantics are isolated from model inference.
- **Scoped change**: Pass. No playback, alteration controls, or export policy is redesigned.

### Post-Design Gate

- The staged candidate never replaces accepted output before validation and commit.
- Pipeline revision prevents superseded work from publishing.
- Previous accepted synthesis and alteration survive same-input failure.
- Fast tests never download the model; the smoke test remains environment-gated.
- No constitutional violations require exception.

## Phase 0: Research

See [research.md](research.md).

Key decisions:

1. Model progress as emitted codec-token count with a 12.5 Hz derived duration.
2. Validate candidate sample rate, non-empty samples, and finiteness at both engine and pipeline boundaries.
3. Stage and atomically replace accepted synthesis output.
4. Use a monotonic pipeline revision to discard superseded results and progress.
5. Preserve prior synthesis and alteration until replacement commits.
6. Keep cancellation out of scope because the upstream blocking generator does not cooperatively cancel.

## Phase 1: Design & Contracts

See:

### Engine Boundary

- `SynthesisRequest`: immutable input snapshot.
- `SynthesisProgress`: emitted token count plus 12.5 Hz estimated duration.
- `SynthesisResult`: sample buffer, rate, and optional statistics with validation.
- `TTSEngine`: load, synthesize, unload; local Sendable values only.

### Pipeline Boundary

- Capture request inputs and current revision before generation.
- Accept progress only for the active revision.
- Validate result, write exclusive staging WAV, recheck revision, then atomically commit.
- Publish synthesis and schedule alteration only after commit.
- Leave prior synthesis/alteration untouched on same-revision failure.

## Implementation Sequence

1. Add typed progress and candidate validation contracts.
2. Use 12.5 Hz progress in Qwen3-TTS adapter and validate adapter output.
3. Add synthesis staging and atomic commit to session files.
4. Add pipeline revision, request snapshots, progress guards, validation, and transactional acceptance.
5. Update smoke output for typed progress.
6. Add fast progress/statistics/output-validation/commit tests.
7. Generate project, build, run focused tests, and refresh Codebase Memory.

## Verification Strategy

1. Run a temporary source-contract verifier from the required macOS temp directory.
2. Generate the Xcode project and verify test membership by matching project lines.
3. Build through XCodeMCP.
4. Run a canonical CLI build.
5. Run focused fast `SynthesisContractsTests` plus non-enabled smoke registration.
6. Do not enable `MVC_TTS_SMOKE` during routine verification.
7. Run `git diff --check`, direct task counts, and a final cleaned ad-hoc verifier.
8. Refresh and verify the persistent Codebase Memory artifact.

## Complexity Tracking

No exceptions. The revision counter, typed progress value, and staged file helper are the minimum mechanisms needed to prevent stale or partial synthesis publication.
