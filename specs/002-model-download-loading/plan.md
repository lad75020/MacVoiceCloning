# Implementation Plan: Model Download and Loading

**Branch**: `feature/time-machine-model-download-and-loading` | **Date**: 2026-07-15 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-model-download-loading/spec.md`

## Summary

Preserve the existing fixed-model lifecycle and native status surface while closing two recovery gaps: completion evidence must be written atomically and observably, and revealing files before the snapshot exists must fall back to the app-owned models directory. Add focused local tests for completion detection without downloading model weights.

## Technical Context

**Language/Version**: Swift 6.0 language mode with Xcode 26 toolchain
**Primary Dependencies**: SwiftUI, Observation, Synchronization, AppKit, Swift Transformers Hub client, Qwen3TTS
**Storage**: User Application Support under the app-owned `Models` directory
**Testing**: Swift Testing through the `MacVoiceCloningTests` Xcode target
**Target Platform**: macOS 26.0+, Apple Silicon
**Project Type**: Native macOS application generated by XcodeGen
**Performance Goals**: Keep lifecycle UI responsive; forward download progress only after at least a 0.5 percentage-point change or completion
**Constraints**: Fixed model identity; multi-gigabyte resumable transfer; no model deletion or in-session cancellation; local loading and synthesis; no broad TTS changes
**Scale/Scope**: One model snapshot and one loaded engine per application session

## Constitution Check

The project constitution is currently an unratified template, so the following repository conventions serve as gates:

- **Native platform behavior**: PASS — retain SwiftUI status controls and Finder reveal behavior.
- **Local data ownership**: PASS — downloads remain under user Application Support.
- **Security/privacy**: PASS — preserve fixed trusted repository identifiers and do not introduce uploads, credentials, or arbitrary remote input.
- **Concurrency safety**: PASS — lifecycle mutation remains main-actor isolated; downloader stays sendable/nonisolated.
- **Testability**: PASS — completion detection is verifiable using a temporary local directory without network or model loading.
- **Build integrity**: REQUIRED — regenerate the Xcode project, build the app, and run focused tests before completion.
- **Scope control**: PASS — only model lifecycle, download completion, reveal behavior, tests, and feature documentation change.

## Phase 0: Research

Research resolves:

1. Why the existing six-state lifecycle remains the compatibility contract.
2. How resumable snapshot transfer and required tokenizer fallback define completeness.
3. Why durable completion evidence must use a throwing atomic write.
4. How Finder reveal behaves before the model-specific directory exists.
5. How to test local completeness without network or loading a multi-gigabyte model.

See [research.md](research.md).

## Phase 2: Implementation Strategy

1. Replace non-throwing completion-marker creation with an atomic throwing write in `App/TTS/ModelDownloader.swift`.
2. Make `App/Model/ModelManager.swift` reveal the model snapshot when it exists and the app-owned models root otherwise.
3. Add `Tests/ModelDownloaderTests.swift` to verify that both tokenizer and completion evidence are required.
4. Regenerate `MacVoiceCloning.xcodeproj` from `project.yml` so the new test is included.
5. Run a focused source-contract verifier, XCodeMCP build, CLI build, and focused test suite.
6. Refresh the persistent Codebase Memory index after verification.

## Verification Strategy

### Automated

1. Temporary source-contract verifier checks state mapping, progress throttling, atomic completion evidence, retry selection, reveal fallback, and status controls.
2. `ModelDownloaderTests` checks incomplete, tokenizer-only, marker-only, and complete local snapshots without network access.
3. XCodeMCP builds the open project first.
4. CLI `xcodebuild` provides a saved build log.
5. `git diff --check` and Spec Kit artifact validation must pass.

### Manual

1. With no complete snapshot, verify identity, approximate size, and Download action.
2. During a real download, verify determinate progress and no conflicting action.
3. With a complete snapshot, verify launch loading, Unload, Load, and Retry transitions.
4. Before and after download, reveal the model location and confirm Finder opens an existing app-owned directory.
